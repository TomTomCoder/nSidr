## Phase 15: AI Provider Gateway

**Goal:** Stand up a Vercel AI SDK in-process gateway as the single AI entry point; manage per-provider keys (with per-model override); expose a modality-filtered model catalog; support Ollama local models; retrofit Phase 17's direct OpenAI calls to route through the gateway.
**Requirements:** GW-01, GW-02, GW-03, GW-04, GW-05
**Success criteria:**

1. User adds an OpenAI (and a second provider) key in settings; a test call succeeds through the gateway.
2. User searches the model catalog and filters by modality (e.g. only image or only text models).
3. User launches/selects an Ollama local model and it answers a prompt without any cloud key.
4. An existing AI feature (e.g. doc embeddings or chat) runs through the gateway with model selectable.
5. Existing direct provider SDK calls in agent execution (Phase 17 baseline) are migrated to route through the gateway ГўВЂВ” no provider SDK imports outside the gateway module.
6. A model can be assigned its own API key, overriding the provider-default key (use case: org has 2 OpenAI keys for cost segregation).

**Plans:** 5/5 plans complete

- [x] 15-02-PLAN.md вЂ” embeddings retrofit through gateway, fail-loud config (GW-04)
- [x] 15-03-PLAN.md вЂ” admin UI: Ollama link+proxy, inline per-model override, embedding picker (GW-02/03/04/05)
- [x] 15-04-PLAN.md вЂ” setting-open-api retrofit: text+image gen through gateway (GW-01/04)

## Phase 17: Agent MCP Enhancement

**Goal:** Extend the existing agent system (memory + table/doc tools) to act on interfaces, invoke plugins, and scope knowledge to selected docs ГўВЂВ” exposed via an MCP layer built on **`@modelcontextprotocol/sdk`** (server + client; AI SDK v6 has no MCP client).
**Status:** Source-complete (2026-06-05) but 2026-06-06 live UAT revealed 6 runtime bugs. 3 critical fixes shipped in commit `00450e8b5`; 3 remain ГўВ†В’ addressed by Phase 17.1.
**Requirements:** AGENT-01, AGENT-02, AGENT-03, AGENT-04, AGENT-05
**Success criteria:**

1. An agent reads/acts on an app or dashboard (not just a table) in a conversation.
2. An agent invokes an installed plugin as a tool and returns its result.
3. An agent answers using only a user-selected set of docs/folders (scoped knowledge search).
4. An agent recalls a fact from a prior conversation (memory) scoped to that agent.
5. Agent tools are served over MCP (verifiable via an MCP client/inspector).

## Phase 24: Agent Runtime Hardening

**Goal:** Fill the five production-readiness gaps identified in the architecture gap analysis (2026-06-14): AI Gateway failover, Input/Output Guardrails, Human-in-the-Loop approval flow, per-user OAuth credential binding, and auto-trigger embedding pipeline.
**Requirements:** ARH-01 (failover), ARH-02 (guardrails), ARH-03 (HITL), ARH-04 (per-user OAuth), ARH-05 (embedding auto-trigger)
**Success criteria:**

1. A configured provider returning HTTP 429 triggers automatic failover to the backup provider; the agent run completes without error.
2. An agent attempting to write an invalid field value (wrong type, fails regex) receives a structured guardrail error as the tool result вЂ” no DB exception propagates.
3. An agent calling `request_human_approval` suspends its run; the UI shows an approval card; accepting resumes the run and the agent continues; rejecting terminates the run with reason.
4. A user-linked Gmail OAuth token is resolved by `(agentId, userId, 'gmail')` and used in preference to the agent-level token.
5. A doc created via `create_knowledge_doc` is indexed and searchable via `search_knowledge_base` within 5 seconds on a local dev stack.
