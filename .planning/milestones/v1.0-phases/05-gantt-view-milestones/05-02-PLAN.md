<objective>
Ensure the NestJS backend accepts Gantt view creation/update, validates that startField and endField reference real date-type fields in the table, and persists all Gantt options correctly.

Purpose: Without backend validation, users can create broken Gantt views pointing to non-date fields, causing frontend errors. This plan adds a validation guard specific to Gantt view options.
Output: view.service.ts handles ViewType.Gantt; view-option-validate.ts enforces date field requirement.
</objective>

  ViewType.Gantt = 'gantt'
  IGanttViewOptions fields: startField (required), endField (required), titleField?, dependencyField?, colorField?, milestoneThreshold (default 0), showCriticalPath (default false), showWeekends (default true), timeScale ('day'|'week'|'month'|'quarter', default 'week')

The view service in NestJS typically:
  - Has a createView(tableId, dto) method
  - Has an updateView(tableId, viewId, dto) method
  - Stores view.options as JSON in the view table
  - May have a switch on view type for type-specific setup

Field types in Teable include 'date' and 'dateTime'. startField/endField must be one of these types.
Prisma field model has: id, tableId, type (FieldType enum from @teable/core).

Pattern for validation: Read field records from prisma, verify type === FieldType.Date or FieldType.DateTime.
