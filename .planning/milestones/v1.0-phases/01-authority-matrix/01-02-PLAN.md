## Trust Boundaries

|----------|-------------|
| Schema definition → consumer (backend + frontend) | Zod schemas are the contract; any drift between schema and runtime causes silent bugs |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | T (Tampering) | createAuthorityMatrixRoleRoSchema.actions field | mitigate | Defined as `z.array(z.string())`; backend (Wave 3 service) MUST cross-check each value against the `Action` enum from `@teable/core` and reject unknown actions with 400. Documented in role-create.ts comment. |
| T-02-02 | I (Info Disclosure) | Vo schemas | accept | All Vo fields are non-sensitive (id, baseId, name, description, actions list, audit timestamps). No tokens, secrets, or PII exposed. |
| T-02-03 | T (Tampering) | name field length | mitigate | All Ro schemas constrain `name` to `.min(1).max(255)` to prevent empty values and excessive payloads. `description` capped at 1000 chars. |
| T-02-04 | S (Spoofing) | createdBy / lastModifiedBy fields in Vo | accept | These are read-only in Vo schemas; service layer (Wave 3) sets them from authenticated session, never from client input. |
</threat_model>
