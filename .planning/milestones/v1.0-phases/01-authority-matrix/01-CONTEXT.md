<domain>
## Phase Boundary

This phase delivers the complete Authority Matrix feature for Teable EE self-hosted:
1. Prisma DB migration — two new tables (`authority_matrix`, `authority_matrix_role`)
2. OpenAPI package — Zod schemas + route configs + axios wrappers for all 8 CRUD endpoints
3. NestJS backend — service, controller, module, registered in app.module
4. Next.js UI — replace the existing stub in `AuthorityMatrix.tsx` with a real management UI
5. AI Interface — verify existing EE activation is functional (no code changes, acceptance-test only)

Runtime enforcement (making the matrix actually gate permissions at request time) is explicitly out of scope for this phase.

</domain>

<decisions>
## Implementation Decisions

### Database Schema (LOCKED)
- Add `AuthorityMatrix` model to `packages/db-main-prisma/prisma/postgres/schema.prisma`:
  - Fields: `id String @id`, `baseId String @map("base_id")`, `name String`, `description String?`, `createdTime DateTime @default(now()) @map("created_time")`, `createdBy String @map("created_by")`, `lastModifiedTime DateTime? @updatedAt @map("last_modified_time")`, `lastModifiedBy String? @map("last_modified_by")`
  - Relation: `base Base @relation(fields: [baseId], references: [id])`
  - Relation: `roles AuthorityMatrixRole[]`
  - `@@index([baseId])`, `@@map("authority_matrix")`
- Add `AuthorityMatrixRole` model:
  - Fields: `id String @id`, `authorityMatrixId String @map("authority_matrix_id")`, `name String`, `actions Json @map("actions")`, same audit fields as above
  - Relation: `authorityMatrix AuthorityMatrix @relation(fields: [authorityMatrixId], references: [id], onDelete: Cascade)`
  - `@@index([authorityMatrixId])`, `@@map("authority_matrix_role")`
- Add back-relation `authorityMatrices AuthorityMatrix[]` to `Base` model
- IDs are NOT `@default(cuid())` — generated by service using `generateAuthorityMatrixId()` (prefix `aut`) and `generateAuthorityMatrixRoleId()` (prefix `aur`) from `packages/core/src/utils/id-generator.ts`

### Migration (LOCKED)
- File path: `packages/db-main-prisma/prisma/postgres/migrations/20260507000000_add_authority_matrix/migration.sql`
- Creates `authority_matrix` and `authority_matrix_role` tables with exact SQL as specified in PRD
- FK: `authority_matrix.base_id` → `base.id` (RESTRICT/CASCADE)
- FK: `authority_matrix_role.authority_matrix_id` → `authority_matrix.id` (CASCADE DELETE)
- Indexes: `authority_matrix_base_id_idx`, `authority_matrix_role_authority_matrix_id_idx`
- After migration: run `pnpm prisma:generate` to regenerate Prisma client

### OpenAPI Package (LOCKED)
- Create directory: `packages/openapi/src/authority-matrix/`
- 8 files: `create.ts`, `get-list.ts`, `get.ts`, `update.ts`, `delete.ts`, `role-create.ts`, `role-update.ts`, `role-delete.ts`, plus `index.ts`
- All follow the pattern from `packages/openapi/src/base/create.ts`: `RouteConfig`, `registerRoute`, `urlBuilder`, axios wrapper function
- Add `export * from './authority-matrix'` to `packages/openapi/src/index.ts`
- Endpoint shapes:
  - POST `/base/{baseId}/authority-matrix` — Ro: `{name, description?}`, Vo: `{id, baseId, name, description?, createdTime, createdBy}`
  - GET `/base/{baseId}/authority-matrix` — Vo: `{list: AuthorityMatrixVo[]}`
  - GET `/base/{baseId}/authority-matrix/{matrixId}` — Vo: matrix with nested `roles: AuthorityMatrixRoleVo[]`
  - PATCH `/base/{baseId}/authority-matrix/{matrixId}` — Ro: `{name?, description?}`, Vo: updated matrix
  - DELETE `/base/{baseId}/authority-matrix/{matrixId}` — 204
  - POST `…/{matrixId}/role` — Ro: `{name, actions: string[]}`, Vo: `{id, authorityMatrixId, name, actions}`
  - PATCH `…/{matrixId}/role/{roleId}` — Ro: `{name?, actions?: string[]}`, Vo: updated role
  - DELETE `…/{matrixId}/role/{roleId}` — 204

### NestJS Backend (LOCKED)
- Files under `apps/nestjs-backend/src/features/authority-matrix/`:
  - `authority-matrix.service.ts` — inject `PrismaService`, `ClsService`; CRUD for matrix and roles; `getByFieldReference(fieldId)` for field-delete integration
  - `authority-matrix.controller.ts` — `@Controller('api/base/')`, all endpoints decorated with `@Permissions('base|authority_matrix_config')` + `@ResourceMeta('baseId', 'params')`
  - `authority-matrix.module.ts` — standard `@Module({ providers, controllers, exports })`
- Register `AuthorityMatrixModule` in `apps/nestjs-backend/src/app.module.ts` imports array
- Field delete integration: extend the backend that handles field deletion references to also query `authority_matrix_role` records and return them in `authorityMatrixRoles` field (per existing `fieldDeleteRefAuthorityMatrixSchema` in openapi)

### Frontend UI (LOCKED)

<specifics>
## Specific Ideas

- Migration timestamp: `20260507000000` (2026-05-07)
- `actions` field stores `string[]` of `Action` values as JSONB — simpler than join table, consistent with other config blobs in the codebase
- `onDelete: Cascade` on `AuthorityMatrixRole` → deleting a matrix auto-deletes its roles
- The nav link (`BasePageRouter.tsx`) is already gated by `base|authority_matrix_config` and the `UpgradeWrapper` — both are bypassed in this EE self-hosted fork (usage returns `Enterprise` level)
- Field delete integration uses the pre-existing `fieldDeleteRefAuthorityMatrixSchema` in openapi — backend just needs to query it

</specifics>
