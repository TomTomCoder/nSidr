<objective>
Implement the backend half of ARH-03 — Human-in-the-Loop (HITL) approval flow. The agent emits a `request_human_approval` tool call that suspends the conversation; on accept the run resumes with the pending action proceeding; on reject the run terminates with the rejection reason. Uses the terminate-and-resume pattern from RESEARCH (Pitfall 2) — NOT in-place generator suspension. Frontend wiring + UAT happen in plan 24-06.

Purpose: Enable agents to gate destructive or sensitive operations behind explicit user approval.
Output: `AgentConversation` schema gains `approvalPayload` + status value; `request_human_approval` tool registered and handled; new POST approve endpoint with ownership check.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → POST /approve | The decision crosses an authenticated session boundary; ownership must be re-checked server-side. |
| LLM → request_human_approval tool input | The `question` and `context` strings are LLM-controlled and will be persisted, then rendered by the UI (plan 24-06) — must remain plain text. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-24-05-01 | Tampering | Approval bypass via direct status update by non-owner | mitigate | `POST /approve` verifies `conversation.userId === req.user.id` (Task 3 Test 3); raw status updates require auth guard. |
| T-24-05-02 | Spoofing | Replayed approval against a non-waiting conversation | mitigate | Endpoint rejects with HTTP 409 unless `status === 'waiting_for_approval'` (Task 3 Test 4); single-use semantics. |
| T-24-05-04 | Denial of Service | Agent emits `request_human_approval` in a tight loop | accept | After approval the run continues; agent prompt-engineering responsibility. Conversation is also bounded by existing per-conversation step limits. |
</threat_model>
