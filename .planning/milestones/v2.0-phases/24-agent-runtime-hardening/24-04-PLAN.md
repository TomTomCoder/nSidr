<objective>
Implement ARH-01 — AI Gateway failover. Today `streamLlmIteration` wraps a single `generateText()` call with no try/catch; any provider error (429, 5xx, network) propagates uncaught and the agent run dies. After this plan, the planner-configured `fallbackModels` list is tried in order on retryable errors, giving zero-downtime LLM dispatch from the agent's perspective.

Purpose: Production resilience — a rate-limited primary provider should not break agent runs.
Output: `aiConfigSchema` gains `fallbackModels?: string[]`; `streamLlmIteration` wraps `generateText` in a for-loop that tries the primary then each fallback on 429/5xx; new spec cases prove retry-on-429 and pass-through-on-4xx behavior.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM provider HTTP boundary | Provider error responses cross into the agent runtime; classifying them dictates retry policy. Misclassification of a 4xx as retryable could amplify bad requests. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-24-04-01 | Denial of Service | Tight retry loop amplifies a non-retryable 4xx into a request storm | mitigate | `isRetryableLlmError` matches only `429` or `>= 500` (or specific network errno codes); 400/401/403/404 short-circuit immediately. Spec Test 3 enforces this. |
| T-24-04-02 | Information Disclosure | Failover masks credential/auth errors (401) that admin needs to see | mitigate | Non-retryable errors propagate unchanged with original status code; admin-facing logs include `modelKey` + `statusCode` per attempt. |
| T-24-04-03 | Tampering | Misconfigured `fallbackModels` includes a provider with mismatched API key | accept | Each fallback resolves via the existing `getModelInstance` path which enforces the provider key registry; a missing key surfaces as a non-retryable config error and propagates. |
</threat_model>
