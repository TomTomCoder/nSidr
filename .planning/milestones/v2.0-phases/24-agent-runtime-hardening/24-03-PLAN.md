<objective>
Implement ARH-02 — input/output guardrails for record-writing tool calls. Today `create_record` writes directly through `recordOpenApiService.multipleCreateRecords` with no schema validation; bad LLM output propagates as a raw Prisma exception that crashes the run rather than letting the LLM self-correct. Also, `update_record` is missing from the switch entirely (per RESEARCH.md Pitfall 1) and must be added under the same guardrail.

Purpose: LLM receives structured `{ success:false, validationErrors:[...] }` on invalid writes and can retry; DB never sees bad values.
Output: New `GuardrailService` using `AbstractField.validateCellValueWithNotNull` from `@teable/core`; both `create_record` and `update_record` switch arms route through the guardrail before commit.
</objective>
