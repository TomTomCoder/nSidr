<objective>
Implement ARH-04 — per-user OAuth credential binding. Today `agent_connection` is keyed on `(agentId, provider)` and `getValidToken(agentId)` returns the single agent-owner token, forcing all users in a base to share one Gmail/Slack/GitHub identity. After this plan, third-party tool calls can bind a token to the *calling user* and the resolver falls back to the agent-level token when no user-scoped row exists.

Purpose: Multi-user orgs can use their own OAuth credentials when an agent invokes Gmail/Slack/GitHub tools, without breaking existing agent-level integrations.
Output: Schema migration adds nullable `userId` to `AgentConnection`; all three OAuth services accept an optional `userId` and apply the documented fallback; `executeToolCall` propagates `ctx.userId` through `executeGmailTool` / `executeSlackTool` / `executeGithubTool`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Agent run context → OAuth credential store | `ctx.userId` (server-authoritative, derived from the authenticated request) flows into the OAuth resolver; never trust a client-supplied `userId`. |
| `agent_connection` row → third-party API | Decrypted OAuth token leaves the boundary as a Bearer header to Gmail/Slack/GitHub. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-24-02-01 | Elevation of Privilege | Cross-user OAuth token access via spoofed userId | mitigate | `ctx.userId` is set by the agent runtime from the authenticated session, never from tool input; OAuth resolver lookup is always scoped to `(agentId, userId, provider)` so a forged userId cannot read another user's token (no token exists at the wrong tuple). |
| T-24-02-02 | Information Disclosure | New per-user rows leak via existing unique constraint collision | accept | `userId` is nullable; existing `@@unique([agentId, provider])` only constrains agent-level rows (userId IS NULL); per-user rows use the new index, no leakage path. |
| T-24-02-03 | Tampering | Migration backfill drops existing agent-level tokens | mitigate | Migration only ADDs a nullable column + index; no UPDATE to existing rows; existing tokens preserved (userId defaults NULL = agent-level). |
</threat_model>
