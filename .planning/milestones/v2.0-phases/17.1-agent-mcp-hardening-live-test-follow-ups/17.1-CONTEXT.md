### D-17.1-01: Bug 3 fix — extend `registerTeableMcpTools` signature
**Approach:** Modify `apps/nestjs-backend/src/features/agent/mcp/teable-mcp-tools.ts:46` to accept an additional `interfaceToolsService: InterfaceToolsService` parameter. Update `tools/list` handler to return `[...registry.getBuiltInTools(), ...interfaceToolsService.getToolDefinitions()]`. Update `tools/call` switch: if `toolName ∈ {get_app, get_dashboard, run_app_action, update_dashboard}` dispatch to `interfaceToolsService.executeInterfaceTool(name, input, {userId, baseId: agentCtx.baseId})`; else fall through to existing `execution.executeTool` path. RBAC: re-use existing `checkPermission()` closure before either dispatch — interface tools already do their own per-action RBAC internally, but the closure is the baseline gate.

**Caller update:** `teable-mcp-server.service.ts:87` `registerTeableMcpTools(...)` call needs to inject `InterfaceToolsService` — add it to the service's constructor (value import per Phase 17 bug-1 lesson; never `import type`).

### D-17.1-02: Bug 4 fix — add new `getEffectiveBaseRole` method, do not mutate existing one
**Approach:** Add `permissionService.getEffectiveBaseRole(baseId: string)` to `apps/nestjs-backend/src/features/auth/permission.service.ts`. Logic: call existing `getRoleByBaseId(baseId)` first; if non-null return it; if null, resolve `baseId → spaceId` via `prisma.base.findUnique({where:{id:baseId}, select:{spaceId:true}})` then call existing `getRoleBySpaceId(spaceId)`; if that returns non-null, **promote** the space role to the base role using `getMaxLevelRole` if it's a base-compatible role.

**MCP gate:** `teable-mcp-server.service.ts` `checkPermission` closure switches from `getRoleByBaseId(agent.baseId)` to `getEffectiveBaseRole(agent.baseId)`. Single line change. `getRoleByBaseId` semantics preserved — no regression risk to REST callers.

**Note:** `AgentPermissionGuard` (REST path) goes through `PermissionGuard` which already inherits space role via a different code path. The new method makes MCP and REST consistent without unifying their resolution paths (which would be a bigger refactor).

### D-17.1-03: Bug 5 / semantic test — vitest integration with stubbed embedding
**Approach:** Add `apps/nestjs-backend/src/features/agent/agent-knowledge-scope.integration.spec.ts`. Pattern: mirror `agent-mcp-flow.integration.spec.ts` — in-memory Prisma stub for `imported_doc` + `doc_chunk`, stub `DocSearchService.hybridSearch` to return chunk fixtures filtered by the scope passed in (so the test asserts the SCOPE was passed correctly, not the embedding ranking).

**Test cases:**
1. Agent with `knowledgeSources={docIds:[A]}` → call `search_knowledge_base` → assert `hybridSearch` called with `scope.docIds=[A]` and result contains only doc-A chunks.
2. Agent with `knowledgeSources={folderIds:[F]}` → assert scope.folderIds=[F].
3. Agent with `knowledgeSources={docIds:[A], folderIds:[F]}` → assert both passed (OR semantics already covered by search.service.ts).
4. Agent with `knowledgeSources=null` → assert hybridSearch called with `scope=undefined` (whole-space search, backward-compat).
5. Out-of-scope doc B exists in fixtures → assert it is NEVER returned in cases 1/2/3.

**Rationale:** Live e2e against real embeddings is non-deterministic and burns API credits; the wiring is what matters and a deterministic test catches the actual bugs (e.g., the executeTool hydrate fix we shipped on 2026-06-06 commit `00450e8b5`).

- ❌ Raw SQL referencing un-quoted camelCase columns OR the wrong table's column name (bug 6)
